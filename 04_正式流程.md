# 04 正式流程

## 当前链路

正式作文批改接口内部流程：

```text
POST /aistudy/api/v2/essay/correct
-> 用户和 token 校验
-> 图片 URL 校验
-> 作文内部意图识别，非作文直接拦截
-> OCR 和视觉转写并发
-> 视觉转写得到整篇作文正文
-> OCR / 物理行切分得到可定位文本行
-> 文本行拼出 sentence_candidates
-> 总评 LLM 生成分数和整体评价
-> 分句点评 LLM 只从 sentence_candidates 里选择问题句
-> 根据 sentence_id / target_text 找回行坐标
-> 生成下划线坐标，做斜率、白带、避让、裁剪
-> 保存历史记录
-> 返回前端结构
```

## 为什么总评和分句点评拆开

总评需要理解整篇文章，重点是内容、结构、语言水平。  
分句点评需要能画线，重点是坐标可靠。

如果都依赖 OCR，会出现整篇作文语义不连贯。  
如果都依赖视觉模型，会出现坐标无法稳定定位。

所以正式方案拆成两条输入：

```text
resolved_essay_text：视觉模型整理后的整篇作文正文，用于总评、评分、历史记录、追问修改。

sentence_candidates：由 OCR/物理行构造的可定位句子候选，用于分句点评和画线。
```

## 分句点评输入

传给分句点评模型的不是整篇 OCR 原文，而是候选句列表：

```json
{
  "sentence_candidates": [
    {
      "sentence_id": "image_001_sentence_001",
      "source_image_index": 1,
      "text": "经过一夜火车的颠簸，我们终于在清晨抵达了山脚。"
    }
  ]
}
```

模型要求：

- 只能从候选句中选择。
- 最多返回 3 条。
- 只评价问题句，不评价亮点句。
- 不评价逗号、句号、标点、不是完整句这类问题。
- `target_text` 尽量保持候选原文，不要自由改写。

## 分句点评输出

内部结果示例：

```json
{
  "sentence_comments": [
    {
      "index": 1,
      "sentence_id": "image_001_sentence_001",
      "target_text": "经过一夜火车的颠簸，我们终于在清晨抵达了山脚。",
      "problem": "这句话交代行程清楚，但表达还可以更有画面感。",
      "polished": "经过一夜火车的颠簸，我们终于在清晨抵达了黄山脚下。"
    }
  ]
}
```

后端再把 `sentence_id` 映射回坐标。

## 返回给前端的坐标

前端只需要使用 `positions`：

```json
{
  "targetText": "经过一夜火车的颠簸，我们终于在清晨抵达了山脚。",
  "problem": "这句话交代行程清楚，但表达还可以更有画面感。",
  "polished": "经过一夜火车的颠簸，我们终于在清晨抵达了黄山脚下。",
  "positions": [
    {
      "sourceImageIndex": 1,
      "x1": 102.3,
      "y1": 482.6,
      "x2": 531.8,
      "y2": 491.2
    }
  ]
}
```

说明：

- `sourceImageIndex` 从 1 开始。
- 坐标基于原图像素。
- 如果前端缩放显示图片，需要按显示比例换算。
- `x1/y1/x2/y2` 可以表示斜线，不一定水平。

## 失败和兜底策略

当前策略偏保守：

```text
候选句不可靠 -> 不让模型评价
模型点评无法定位 -> 过滤掉该点评
坐标压字严重 -> 尝试白带避让
避让仍失败 -> 不返回该坐标
非作文图片 -> 直接拦截，不进入批改
```

这样做的原因是：少一条点评可以接受，画到完全不相关的位置不能接受。

## 后续可优化点

1. 建立固定回归集，每次修改自动生成对比图。
2. 给每条下划线增加内部置信度，低置信度不展示。
3. 对模糊图增加更强的图像增强策略。
4. 对多图作文增加更明确的页序校验。
5. 将测试报告自动生成成 HTML，方便人工快速检查。
